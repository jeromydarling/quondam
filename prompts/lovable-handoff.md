## Task 1 — Replace `src/auth/source.ts`

Add `@supabase/supabase-js` as a dependency. Configure Supabase env vars in `.env.local`:

```
VITE_SUPABASE_URL=https://YOUR_PROJECT.supabase.co
VITE_SUPABASE_ANON_KEY=YOUR_ANON_KEY
```

Replace the contents of `src/auth/source.ts` with a class implementing the existing `AuthBackend` interface. **Do not change the interface** — every consumer in the app imports `authBackend` from this one path.

The interface, copied verbatim so you don't have to grep:

```typescript
import {
  type AuthResult,
  type User,
  err,
  ok,
} from "./types";

export interface AuthBackend {
  getCurrentUser(): Promise<User | null>;
  onAuthChange(cb: (user: User | null) => void): () => void;
  signUpWithEmail(email: string, password: string): Promise<AuthResult>;
  signInWithEmail(email: string, password: string): Promise<AuthResult>;
  signInWithGoogle(): Promise<AuthResult>;
  signOut(): Promise<void>;
  changePassword(oldPassword: string, newPassword: string): Promise<AuthResult<true>>;
  changeEmail(newEmail: string, password: string): Promise<AuthResult>;
  resendVerification(): Promise<AuthResult<true>>;
  markVerified(): Promise<AuthResult>;
  resetPasswordRequest(email: string): Promise<AuthResult<true>>;
  deleteAccount(password: string): Promise<AuthResult<true>>;
}

export const authBackend: AuthBackend = /* your SupabaseAuthBackend instance */;
```

The `User` shape (defined in `src/auth/types.ts`):

```typescript
interface User {
  id: string;             // map to supabase user.id
  email: string;
  emailVerified: boolean; // supabase user.email_confirmed_at !== null
  createdAt: string;      // supabase user.created_at, ISO
  authProvider: "email" | "google";
}
```

`AuthResult` is a discriminated union:

```typescript
type AuthResult<T = User> =
  | { ok: true; value: T }
  | { ok: false; error: { code: AuthErrorCode; message: string } };
```

Map Supabase errors to the `AuthErrorCode` union:
`"invalid-credentials" | "email-in-use" | "weak-password" | "invalid-email" | "not-signed-in" | "user-not-found" | "wrong-password" | "network" | "unknown"`

### Behavior notes

- **Email verification**: do **not** enforce a wall. Sign-in must work immediately for unverified accounts. The existing UI shows a banner until `emailVerified` is true. Configure your Supabase Auth project to allow sign-in before email confirmation.
- **`markVerified()`** exists for the dev stub. With Supabase the user clicks the email link and Supabase handles it. Implement `markVerified()` as a `getCurrentUser()` refresh — if Supabase reports the email as confirmed, return the new user; otherwise return the unchanged one. The "I've verified" button in the UI calls this method to re-pull state.
- **Google OAuth**: enable Google as an auth provider in the Supabase dashboard. Use `supabase.auth.signInWithOAuth({ provider: 'google' })`. The redirect URL is the deployed app root.
- **`onAuthChange`**: subscribe via `supabase.auth.onAuthStateChange`. Return an unsubscribe function. Fire once with the current state on subscribe so the existing AuthContext hydrates.
- **`resetPasswordRequest`**: call `supabase.auth.resetPasswordForEmail(email, { redirectTo: '<app>/#/sign-in' })`.
- **`deleteAccount`**: Supabase doesn't allow client-side account deletion by default. Either (a) call a Supabase Edge Function you create that uses the service role key to delete the user, or (b) call `supabase.auth.admin.deleteUser` from a Function. This is the only piece of this task that requires server-side code; create the smallest Function that does the job.
- **Plaintext is gone**: Supabase handles password hashing. The existing stub stores plaintext in localStorage — that's why it must be replaced.

## Task 2 — Replace `src/catalog/source.ts`

The current file is:

```typescript

export async function loadCatalog(): Promise<Catalog> {
  return catalogJson as Catalog;
}
```

Replace with a Supabase query that returns a `Catalog` of the **same shape**. Cache the result in module scope so we only fetch once per session.

The `Catalog` and `Story` types live in `src/catalog/types.ts` — read that file, do not redefine the types.

### Database schema

Create a `stories` table in Supabase with one column per Story field. Columns that are objects (`source`, `loudness`, `safety`, `restoration`, `series`) can be `jsonb`. Columns that are arrays (`mood`, `chapters`) can also be `jsonb` (or `text[]` for `mood`). The `id` column is `text PRIMARY KEY` — these IDs are stable forever (used as keys for resume positions and favorites).

Seed the table from the existing `catalog/catalog.json` (6 entries today, more to come from the curator pipeline). The exact JSON shape is enforced by `catalog/schema.json` — use it as the source of truth for column types.

Add a `version` and `generated_at` row to a tiny `catalog_meta` table or just hardcode `version: "1"` and use `now()` for `generatedAt` in the response.

### Row-level security

Enable RLS on `stories` and create a single policy: `select` is allowed for everyone (anon + authenticated). No insert/update/delete from the client — curators write via the existing `catalog/catalog.json` PR flow until that pipeline migrates.
